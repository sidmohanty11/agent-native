# 人在環核准

大多數 actions 應該直接執行。其中一些操作（傳送電子郵件、為卡充值、刪除帳戶）是面向外部且難以復原的，您不希望代理自主執行這些操作。對於這些，`defineAction` 有一個選取加入的**核准門**：當代理嘗試呼叫該操作時，迴圈會暫停，向人類顯示核准/拒絕功能，並*僅*在人類核准該特定呼叫後才執行該操作。

> [!WARNING]
> 保持很少的核准。每個門控操作都是代理迴圈中的硬停止——它會中斷執行並需要人工往返。僅將 `needsApproval` 用於真正後果嚴重、難以撤銷、面向外的操作。如果您發現自己對讀取或例行寫入進行門控，那麼您就錯了。預設為**關閉**，幾乎每個操作都應將其關閉。

## `needsApproval` 門 {#needs-approval}

在 `defineAction` 上設定 `needsApproval`。它接受布爾值或謂詞：

- **`needsApproval: true`** — 始終需要核准。
- **`needsApproval: (args, ctx) => boolean | Promise<boolean>`** — 僅當謂詞返回 true 時才需要核准。有條件地選取門，例如僅適用於外部收件人或僅高於美元閾值：

  ```ts
  needsApproval: (args) => !args.to.endsWith("@your-company.com"),
  ```

  保持謂詞純粹且快速。 **關閉失敗**：如果謂詞拋出異常，框架會將其視為“需要核准”，而不是默默地執行後果嚴重的操作。

當 `needsApproval` 被省略時，行為是逐位元組不變的——公開路徑上沒有額外的成本。

這對於舊版 `parameters` 樣式的 actions 和基於架構的 actions，以及應用內代理、子代理、A2A 和 MCP 呼叫者（每個代理透過同一迴圈進行表面路由）的工作原理相同。

## 迴圈如何暫停 {#loop}

當代理呼叫門控操作並且此特定呼叫尚未獲得核准時，迴圈不會執行 `run()`。相反，它：

1. 解析門。對於謂詞，它呼叫`needsApproval(input, ctx)`；拋出被視為“必須核准”（失敗關閉）。
2. 發出 `tool_start` 事件（以便 UI 顯示呼叫），然後立即發出 **`approval_required`** 事件，然後停止轉彎。該操作的副作用永遠不會發生。

`approval_required` 事件包含用戶端呈現可供性所需的一切：

| 欄位          | 型別     | 註解                                         |
| ------------- | -------- | -------------------------------------------- |
| `tool`        | `string` | 代理嘗試呼叫的操作名稱。                     |
| `input`       | 物件     | 代理傳遞的參數。                             |
| `approvalKey` | `string` | **穩定金鑰**用戶端回顯以核准*此確切的呼叫*。 |
| `toolCallId`  | `string` | 模型端工具呼叫 ID（如果可用）。              |

`approvalKey` 是根據工具名稱及其輸入確定性派生的，因此相同的邏輯呼叫始終會生成相同的金鑰。模型永遠不會看到或設定它 - 它純粹是框架和人類的核准功能之間的握手。

暫停工具返回一個結果，告訴模型轉動已暫停並且不再重試，因此模型不會旋轉。

## 人類如何認可 {#approve}

在 `approval_required` 上，聊天 UI 在暫停的工具呼叫上呈現 **核准/拒絕** 提示。這是在 `AssistantChat` 中自動連線的 - 您無需根據範本建置它。

- **核准**重新發出在 `approvedToolCalls: [approvalKey]` 中攜帶呼叫金鑰的輪次（普通的繼續訊息）。在重新發出的回合中，門會看到已核准的集合中的金鑰，並讓特定的呼叫正常執行。
- **Deny** 在本機消除可供性；沒有重新發出任何內容，因此該操作永遠不會執行。

`approvedToolCalls` 是聊天請求 (`AgentChatRequest.approvedToolCalls`) 上的欄位。其中不存在的金鑰將保持暫停狀態 - 核准一個呼叫永遠不會茫然地核准其他呼叫。因為金鑰是內容尋址的，所以核准授權*使用這些參數進行呼叫*；如果模型稍後建議不同的傳送，那就是新的金鑰和新的核准。
